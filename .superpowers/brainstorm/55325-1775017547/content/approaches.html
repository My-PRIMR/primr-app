Three approaches to creator results tracking

All three deliver the same features. The difference is architecture and where the data fetching lives.

A

Pure server-rendered (RSC) Recommended

Results tab and lesson drilldown are both Next.js Server Components. Data fetched at request time with Drizzle queries. No new API routes.

Pros

  • No client JS for data
  • Consistent with existing pattern (my-primr page)
  • Simpler — less code

Cons

  • No interactive sorting/filtering without extra client component
  • Full page reload to change filters
B

API-first (client components)

New API routes return JSON. Creator dashboard fetches client-side. Enables real-time sorting, filtering, pagination without page reloads.

Pros

  • Interactive table sorting/filtering
  • Can add pagination easily

Cons

  • More code (API routes + client fetching)
  • Two different data patterns in the app
C

Hybrid — RSC summary, client drilldown

Results tab is server-rendered (aggregate totals). Per-lesson drilldown is a client component with an API route — enables sorting the learner roster by score/date.

Pros

  • Right tool for each job
  • Interactive where it matters (drilldown)

Cons

  • Two patterns to maintain
  • Slightly more complexity

My recommendation: A — pure RSC keeps it simple and consistent. The learner roster table can still be sorted via URL search params (a Next.js pattern), and we can add client interactivity later if needed.
